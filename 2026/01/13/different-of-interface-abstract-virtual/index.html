different of interface, abstract, virtual.

  • abstract class(抽象類別)？

    1. 在類別宣告中使用 abstract 修飾詞，來表示某一類別只是要作為其他類別的基底類別，不是自行具現化(不可以被newe)。要繼承的子類別需要共用欄位/屬性/方法實作（如 Driver、Options 建立與釋放、共用工具方法），可使用abstract class，讓子類別繼承共用邏輯，只需實作差異化部分（如 RunAsync），可避免重複程式碼，維護性高。
    2. abstract class中可以包含抽象成員（沒有實作，且必須在子類別中被覆寫）和完全實作的成員（例如一般方法、屬性和建構函式）。
    3. abstract class 裡的 abstract method，子類別「一定要」用 override 實作，否則直接編譯失敗。

  • Virtual & Override

    1. 當abstract class裡的method為virtual簽章時，繼承的子類別可以透過override簽章來覆寫父類別的virtual method內的實作。子類別也可以透過base來直接呼叫父類別的virtual method實作。

  • Interface

    1. 只定義合約（方法/屬性），不提供任何實作。
    2. 讓不同類型的 crawler 可以有完全不同的實作（ex： 由Selenium 替換成 Playwright，不依賴特定class，達到解藕目的。）。
    3. 適合只需規範行為、無共用邏輯時。

程式範例(網路爬蟲)

  • Interface：定義「我是一個爬蟲」
    上層模組只依賴 interface
    • 不管爬蟲套件是 Selenium / Playwright
    • 不管實作細節
    • DI / 排程 / Mock 都靠它

  • Abstract class：定義共用流程(邏輯)骨架
    • 實作 interface
    • 提供可 override 的擴充點

  • 子類別實作（override 使用範例）
    Site A：只實作「必須的 abstract」
    • CrawlCoreAsync為abstract成員，必須強制override。

    Site B：override virtual（可選）

    Site C：override + 呼叫 base（進階）

    上層使用（只看 interface）
    • 完全不知道 Selenium，只依賴於interface，可把實作隨時替換成Playwright達成與Selenium解藕。
    • 提供可 override 的擴充點
